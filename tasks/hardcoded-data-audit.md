# Hardcoded / Duplicated / Stale-Prone Data Audit

Read-only audit. No code changed. Scope: `index.html`, `scripts/*.mjs`, `scripts/lib/*.mjs`, `data/*.js`, plus `api/*.js` (Supabase). Excluded per request: movement-palette hexes, and the pool/teach/hotspot data files themselves.

Legend for **Risk**: drift = two copies can silently diverge; stale = hand list breaks/rots when the pool changes; secret = credential sprawl.

## 2. Highest-value single-sourcing opportunities (do these first)

1. **Collections metadata (#3, #10, #14).** One change kills three duplications: delete the inline `COLL_META`/`COLL_VIA_WD`/`COLL_CANON`/`COLL_BUCKET`/`computeCollections`/`era`/`by` from index.html and render the already-loaded `window.ARTEFACTUM_COLLECTIONS` (generated by `make-collections.mjs`). The generated file is loaded but unused today, and the inline copy has already drifted (`loc` source, extra buckets). Biggest dup-surface removed for the least work. Caveat: `computeCollections` runs against the *live* `POOL` (which can include the day's frozen set), whereas the generated file is a build snapshot ‚Äî confirm a build-time snapshot is acceptable for the Collections page (it should be; counts are descriptive).

2. **Supabase config (#2).** Set `SUPABASE_URL` + `SUPABASE_ANON_KEY` in Vercel env, strip the 4 anon-key literals and 8 URL literals from `api/*.js` (the `process.env.X || '...'` fallbacks already exist ‚Äî just remove the fallback string). Leaves exactly one client-side literal in index.html (unavoidable without a build).

3. **MOVEMENTS ‚Üí generated, pool-gated (#1).** The single most stale-prone object in the app: 58% of pool styles fall through to a generic emblem and "c. unknown" dates, and 13 entries are dead. Generate a `data/movements.js` skeleton from the pool, keep only `dates` as hand data, derive `region` from `styleRegion`, and add a `check-pool`-style gate that fails when a pool style has no dates.

4. **Demonym unification (#4).** Collapse `DEMONYM` + `TRAIN_DEMONYM` into one adjective‚Üícountry map keyed to `data/countries.js` names; derive the short culture token.

## 3. index.html ‚Üî scripts/lib copies ‚Äî keeping them in sync without a build

These are copies **by necessity** (index.html loads no modules, has no build step). Catalogue + current state:

| style canonicalization (`CANON_STYLE` vs `canonicalizeStyle`/`STYLE_MERGE`) | 620‚Äì622 | domain.mjs:18‚Äì31 | **No** | Overlapping, separately maintained |
| `normalizeArtist` / artist-match | norm/artistMatch inline | domain.mjs:10‚Äì15 | No | Different shapes (client `norm` is its own thing) |

**Recommendation ‚Äî extend the proven `medium.test.mjs` pattern (a diff-test), not a generate step.** A generate-step (lib ‚Üí inject into index.html) is the cleaner ideal, but it introduces the build the app deliberately avoids and risks corrupting the 2,200-line single file. The regex-extract diff test is already trusted, in CI, and zero-build. Concretely:

1. **Add structural-equality assertions** for the copied *data objects* (`MED_FAMILY`, and `MED_FAMILY` keys vs `ALL_MED`) ‚Äî extract the object literal from index.html, `JSON.parse`-ish via `new Function`, and `deepEqual` against the lib export. Catches object drift the fixture test can miss.
2. **Add a style-canonicalization diff test** mirroring `medium.test.mjs`: extract `CANON_STYLE` from index and assert it's a subset of / consistent with `canonicalizeStyle`. Better: make the pipeline canonicalize styles so `CANON_STYLE` becomes empty (then assert it's empty).
3. **Document the contract** in a one-line comment over each client copy: `// COPY of scripts/lib/<file>.<fn> ‚Äî kept in sync by tests/<x>.test.mjs; edit both.` (domain.mjs already has this note; index.html copies mostly don't.)
4. **Pool-coverage gates** (new, distinct from drift): tests that fail when a pool style has no MOVEMENTS dates (#1), no MOV_FAMILY (#8), or no CUE_ALIASES/cues (#9) ‚Äî these catch the stale-list class, which no current test covers.

Net: one new diff test file + extend `npm test`. No build step, consistent with the existing approach and the user's "fail-closed gate in npm test" preference.
